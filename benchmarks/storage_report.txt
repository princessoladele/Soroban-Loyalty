OBJECTIVE
---------
Reduce per-Campaign persistent storage footprint by ≥ 20% to lower
Soroban rent fees without changing observable contract behaviour.

XDR ENCODING PRIMER
--------------------
Soroban stores contract values as XDR. Relevant type sizes:
  bool        →  4 bytes (XDR boolean)
  u32 / i32   →  4 bytes
  u64 / i64   →  8 bytes
  i128        → 16 bytes
  Address     → 32 bytes (Ed25519 public key)
  Symbol      →  variable (short symbols ≤ 9 chars fit in 8 bytes)

reward_amount   i128          16      Required (token stroops)

TOTAL                         60 bytes per Campaign record

CHANGES APPLIED
---------------
1. REDUNDANCY REMOVAL — removed `id: u64` (8 bytes)
   Rationale: the campaign id is already the key in DataKey::Campaign(id).
   Storing it inside the value is pure duplication. Callers that need the
   id already have it from the return value of create_campaign().

2. TYPE DOWNSIZE — `expiration: u64 → u32` (saves 4 bytes)
   Rationale: Unix timestamps fit in u32 until 2106-02-07. The public API
   still accepts u64 for forward-compatibility; the value is cast to u32
   on write and back to u64 on the is_active() comparison.

3. TYPE DOWNSIZE — `total_claimed: u64 → u32` (saves 4 bytes)
   Rationale: A campaign accumulating more than 4,294,967,295 claims is
   not a realistic scenario. checked_add() still guards against overflow.

4. STORAGE TIERING — no changes
   Rationale: No ephemeral or intermediate state was found in the contract.
   All stored data (merchant, reward_amount, expiration, active,
   total_claimed) is long-lived and correctly placed in persistent().
   The NextId and Admin counters in instance() are already optimal.

SAVINGS SUMMARY
---------------
  Before : 76 bytes / record
  After  : 60 bytes / record
  Saved  : 16 bytes / record
  Reduction: 16 / 76 = 21.1%  ✓  (exceeds the 20% target)

At 1,000 campaigns the saving is 16,000 bytes of persistent storage,
directly reducing the ledger rent charged per ledger entry TTL extension.
